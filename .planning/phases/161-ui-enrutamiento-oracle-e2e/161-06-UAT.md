### 3. Oracle 3 (VER-03) — `reasoning_tokens > 0` logged to JSONL

expected: Next CatBot message after VER-02 succeeds triggers `logger.info('catbot-chat', 'reasoning_usage', {reasoning_tokens > 0, alias: 'catbot', ...})` per Plan 161-03 instrumentation
result: issue
reported: "First prompt errored with LiteLLM 400 (collision between `thinking` and `thinking_level`); second prompt (with explicit `model: claude-opus` in body) returned a valid response with reasoning but NO `reasoning_usage` JSONL line ever appeared. `grep -ac 'reasoning_usage' /app/data/logs/app-2026-04-22.jsonl` → 0."
severity: blocker

**Prompt (verbatim):**
> Resuelve razonando paso a paso: dos trenes parten de estaciones A y B separadas 600 km. El que sale de A viaja a 80 km/h, el que sale de B viaja a 70 km/h. Salen simultáneamente y viajan uno hacia el otro. ¿A qué hora se cruzan y a cuántos km de A?

**Anomaly chain:**

**Attempt 1** (no explicit `model` in request body, relying on `catbot_config` + alias):

LiteLLM returned HTTP 400:
```
litellm.UnsupportedParamsError: Cannot specify both `thinking` (which maps to
`thinking_budget`) and `thinking_level` in the same request. For Gemini 3
models, use `thinking_level` instead. Received Model Group=gemini-main
```

**Critical observation:** `/api/catbot/chat` log says `"Alias resolved: catbot -> claude-opus"` (correct per alias state), but LiteLLM reports `Model Group=gemini-main` (wrong model actually hit the gateway). Root cause → Gap A below.

**Attempt 2** (with `"model":"claude-opus"` injected into request body, bypassing `catbot_config.model`):

CatBot returned a valid step-by-step reasoning response (~820 output tokens with proper kinematic derivation: `t = 600 / (80+70) = 4h`, cross at `4·80 = 320 km from A`). But the expected `reasoning_usage` JSONL line **never appeared in the log**:

```bash
$ docker exec docflow-app grep -ac "reasoning_usage" /app/data/logs/app-2026-04-22.jsonl
0
```

**Control test (direct bypass of DocFlow):**

To confirm LiteLLM passthrough is in fact working, a direct curl to LiteLLM with the same body shape:

```
POST /v1/chat/completions
{
  "model": "claude-opus",
  "max_tokens": 32000,
  "reasoning_effort": "high",
  "thinking": {"type": "enabled", "budget_tokens": 16000},
  "messages": [...]
}
```

Response contained:
- `usage.completion_tokens_details.reasoning_tokens = 50` (non-zero ✓)
- `message.reasoning_content = "<Claude thinking text>"` ✓
- `message.thinking_blocks = [{type: "thinking", thinking: "...", signature: "..."}]` ✓

**Interpretation:** LiteLLM passthrough (Phase 159-02/04) is correctly implemented and does return `reasoning_tokens`. But the end-to-end flow from `/api/catbot/chat` to the Plan 161-03 logger does not fire for reasons that are not yet diagnosed. See Gap B.

**VER-03 contract:** NOT satisfied. The logger is the canonical evidence mechanism for VER-03 (`{source:"catbot-chat", message:"reasoning_usage", reasoning_tokens:>0, alias:"catbot"}`) — without the JSONL entry, the truth is unverifiable per the hard spec.
