Create `app/src/lib/services/canvas-executor.ts` mirroring `task-executor.ts` patterns. Full implementation:

1. **Types and state**: Define `CanvasNode`, `CanvasEdge`, `NodeStatus` ('pending'|'running'|'completed'|'failed'|'waiting'|'skipped'), `NodeState`, `NodeStates`. Create `runningExecutors = new Map<string, { cancelled: boolean }>()`.

2. **export function topologicalSort(nodes, edges)**: MUST be exported (used by validate route in Task 2). Kahn's algorithm (~20 lines). Build inDegree map + adjacency list from nodes/edges. Start with 0-inDegree nodes in queue. Process queue: push to order, decrement neighbors' inDegree, enqueue when 0. Return string[] of node IDs.

3. **callLLM(model, systemPrompt, userContent)**: Copy from task-executor.ts. Fetch to `process['env']['LITELLM_URL'] || 'http://litellm:4000'` + `/v1/chat/completions`. Parse response for output, input_tokens, output_tokens. Return `{ output, tokens, input_tokens, output_tokens, duration_ms }`.

4. **getRagContext(projectId, query, maxChunks)**: Copy from task-executor.ts. Fetch embedding from ollama, search qdrant collection `project_{projectId}`, return concatenated chunk texts.

5. **dispatchNode(node, edges, nodeStates, canvasId, runId)**: Switch on node.type:
   - `start`: Return `{ output: node.data.initialInput || '' }` — passthrough, no LLM.
   - `agent`: Get predecessorOutput. Call callLLM with `node.data.model || 'gemini-main'`, system prompt using node.data.instructions, user content = predecessorOutput. If node.data.useRag and node.data.agentId, prepend RAG context. Log usage per call with `logUsage({ event_type: 'canvas_execution', agent_id: node.data.agentId, model, ... })`.
   - `project`: Get predecessorOutput. Call getRagContext with node.data.projectId, node.data.ragQuery || predecessorOutput, node.data.maxChunks || 5. Return RAG results as output. No LLM call.
   - `connector`: Get predecessorOutput. Fetch connector config from DB. If mode='before', call connector URL with payload THEN return predecessor output. If mode='after', return predecessor output (connector fires after). Use `fetch()` for HTTP connectors. For n8n_webhook, POST to webhook URL with payload.
   - `checkpoint`: Return `{ output: predecessorOutput }` — the executor main loop handles the 'waiting' status transition separately (see research Pattern 9).
   - `merge`: Collect all incoming edges' source outputs from nodeStates. If node.data.agentId, call callLLM to synthesize. Otherwise concatenate with separators. Log usage if LLM called.
   - `condition`: Call callLLM with fast model ('gemini-main'), system prompt "Eres un evaluador de condiciones. Responde SOLO con 'yes' o 'no'.", user content with the condition text + predecessorOutput. Parse response. Return `{ output: 'yes' | 'no' }`.
   - `output`: Passthrough — return predecessorOutput. Apply format if node.data.format is 'json' (try JSON.parse/stringify prettify, fallback to raw).

6. **getPredecessorOutput(nodeId, edges, nodeStates)**: Find incoming edge, return source node's output from nodeStates.

7. **getSkippedNodes(conditionNodeId, chosenBranch, nodes, edges)**: Find the sourceHandle edge that was NOT chosen (e.g., if chosen='yes', find edge with sourceHandle='no'). DFS from that edge's target node, collecting all reachable node IDs. BUT stop and do NOT skip a node if it has another incoming edge from a non-skipped node (MERGE convergence case).

8. **executeCanvas(canvasId, runId)**: Main execution loop (see research code example). For each nodeId in executionOrder: skip if status='skipped'; mark as 'running'; call dispatchNode; handle checkpoint (set 'waiting', update run status, return); handle condition (mark skipped branch); mark 'completed' with output/tokens; check cancelled flag. On completion, update run status to 'completed', log total usage. On error, mark node 'failed' and run 'failed'.

9. **cancelExecution(runId)**: Set cancelled flag in runningExecutors Map. Update DB status to 'cancelled'.

10. **resumeAfterCheckpoint(runId, checkpointNodeId, approved, feedback?)**: If approved: mark checkpoint 'completed', update run status to 'running', call executeCanvas (fire-and-forget). If rejected: find predecessor via incoming edge, reset predecessor to 'pending' with feedback in its state, reset checkpoint to 'pending', call executeCanvas.

11. **saveNodeStates(runId, currentNodeId, nodeStates)**: Helper to UPDATE canvas_runs SET node_states=?, current_node_id=?.

**Modify usage-tracker.ts**: Add `'canvas_execution'` to the UsageEvent.event_type union type.

**Modify db.ts**: In the startup cleanup section, add `db.prepare("UPDATE canvas_runs SET status = 'failed' WHERE status = 'waiting'").run();` alongside the existing 'running' cleanup.

IMPORTANT patterns:
- Use `process['env']['LITELLM_URL']` bracket notation (not dot notation)
- Use `process['env']['OLLAMA_URL']` and `process['env']['QDRANT_URL']` bracket notation
- Use `generateId()` for run IDs, NOT `crypto.randomUUID()`
- All text in Spanish where user-facing (error messages in API responses)

**1. POST /api/canvas/[id]/execute/route.ts** (EXEC-01):
- Read canvas from DB, validate flow_data exists
- Parse flow_data, extract nodes/edges
- Run topologicalSort (import from canvas-executor or inline the same function)
- Initialize all node states as `{ status: 'pending' }`
- Create canvas_run row with generateId(), status='running', node_states JSON, execution_order JSON, started_at
- Fire-and-forget: `executeCanvas(params.id, runId).catch(console.error)`
- Return `{ runId, status: 'running' }`

NOTE: The topologicalSort function should be exported from canvas-executor.ts (add `export` to it in Task 1 if not already done) and imported here, NOT duplicated.

**2. GET /api/canvas/[id]/run/[runId]/status/route.ts** (EXEC-02):
- Query canvas_runs WHERE id=runId AND canvas_id=id
- Parse node_states and execution_order from JSON
- Calculate elapsed_seconds from started_at
- Count completed_steps (status 'completed' or 'skipped')
- Return: `{ status, node_states, current_node_id, execution_order, total_steps, completed_steps, elapsed_seconds, total_tokens, total_duration, completed_at }`

**3. POST /api/canvas/[id]/run/[runId]/checkpoint/[nodeId]/approve/route.ts** (EXEC-07):
- Validate run exists and status is 'waiting'
- Call `resumeAfterCheckpoint(runId, nodeId, true)`
- Return `{ ok: true }`

**4. POST /api/canvas/[id]/run/[runId]/checkpoint/[nodeId]/reject/route.ts** (EXEC-08):
- Parse body for `{ feedback }` (optional textarea content)
- Validate run exists and status is 'waiting'
- Call `resumeAfterCheckpoint(runId, nodeId, false, feedback)`
- Return `{ ok: true }`

**5. POST /api/canvas/[id]/run/[runId]/cancel/route.ts** (EXEC-10):
- Call `cancelExecution(runId)` from canvas-executor.ts
- Return `{ ok: true }`

All routes follow existing patterns from `/api/tasks/` routes. Use `NextResponse.json()` for all responses. Use try/catch with 500 error responses. Route params accessed via `{ params }: { params: { id: string; runId: string; nodeId?: string } }`.
