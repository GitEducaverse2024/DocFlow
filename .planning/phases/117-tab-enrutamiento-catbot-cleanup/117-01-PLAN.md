  <action>
Create `tab-enrutamiento.tsx` as a 'use client' component that:

1. **Data fetching on mount** with Promise.all for 3 endpoints:
   - GET /api/aliases -> aliases array
   - GET /api/mid?status=active -> MID models array
   - GET /api/models/health -> health result (for alias statuses)
   Store in state: aliases, midModels, healthResult, loading flag.

2. **Compact table layout** (NOT cards like other tabs -- a table is more appropriate for routing):
   - Columns: Alias (font-mono), Modelo (dropdown), Estado (semaphore), Tier (badge)
   - Each row: dark bg-zinc-950/40 with zinc-800 border, compact padding
   - Mobile: stack columns vertically with flex-col on small screens

3. **Health semaphore per alias** using AliasHealth from health data:
   - Match alias name to healthResult.aliases[].alias
   - direct -> green dot (bg-emerald-500) + "Directo" text
   - fallback -> amber dot (bg-amber-500) + "Fallback" text
   - error -> red dot (bg-red-500) + "Error" text
   - no health data -> gray dot (bg-zinc-600) + "Sin datos" text

4. **Model dropdown** using shadcn Select:
   - Show ALL MID models (active status) in the dropdown
   - For each model, check availability: compare model_key against healthResult.providers or use a simple availability check -- model is available if ANY alias resolves to it or if its provider has status 'connected' in health data
   - Available models: normal text style
   - Unavailable models: text-zinc-600 (grayed out) + small AlertTriangle icon (amber) + "(no disponible)" suffix
   - Current model_key shown as selected value
   - Do NOT use isModelAvailable from catbot-tools.ts (that's server-side). Instead, derive availability client-side: a model is "available" if its provider appears in healthResult.providers with status 'connected'. Build a Set of connected providers from health data, then check if model's provider field is in that set.

5. **Pre-change availability verification** (ROUTING-04):
   - When user selects a new model from dropdown:
     a. If model's provider is in connectedProviders set -> proceed with PATCH
     b. If model's provider is NOT connected -> show a confirmation toast/dialog: "El modelo {model} no esta disponible actualmente. Cambiar de todas formas?"
     c. On confirm or if available -> PATCH /api/alias-routing with { alias, model_key: newKey }
     d. Optimistic update: update aliases state immediately, revert on error
     e. Show toast on success/error via sonner

6. **Tier badge** per alias row:
   - Look up the alias's current model_key in midModels to find tier
   - Use getTierStyle() from '@/lib/ui/tier-styles' for badge coloring (same pattern as other tabs)
   - If model not in MID: show "Sin ficha" amber badge (same as existing alias-routing-table.tsx pattern)

7. **i18n keys** under settings.modelCenter.enrutamiento namespace:
   - Spanish (es.json): alias, modelo, estado, tier, sinDatos, directo, fallback, error, sinFicha, noDisponible, confirmarCambio, cambiar, cancelar, updated, updateError, loading, noAliases
   - English (en.json): same keys in English

Use useTranslations('settings.modelCenter.enrutamiento').

IMPORTANT conventions:
- Use process['env'] bracket notation if any env vars needed (unlikely here)
- All visible text via i18n keys, no hardcoded strings
- Dark theme colors per conventions (bg-zinc-950, cards bg-zinc-900/80, borders zinc-800)
- Loader2 spinner while loading
