**Paso 2: Analizar resultados por nodo**

Consultar el canvas_run mas reciente:
```bash
cd /home/deskmath/docflow/app && node -e "
const Database = require('better-sqlite3');
const db = new Database('data/docflow.db');
const run = db.prepare(\"SELECT * FROM canvas_runs WHERE canvas_id = '{canvas_id}' ORDER BY created_at DESC LIMIT 1\").get();
console.log('Status:', run.status);
const states = JSON.parse(run.node_states || '{}');
for (const [nodeId, state] of Object.entries(states)) {
  console.log(\`\nNode: \${nodeId}\`);
  console.log(\`  Status: \${state.status}\`);
  console.log(\`  Output: \${JSON.stringify(state.output)?.substring(0, 500)}\`);
  if (state.error) console.log(\`  ERROR: \${state.error}\`);
}
db.close();
"
```

**Paso 3: Verificar criterios por nodo**

Para cada nodo, verificar el output:
- **Normalizador**: JSON valido con 6 campos (from, subject, date, body_plain, has_attachments, message_id)
- **Clasificador**: producto correcto (K12/REVI/spam), template_id correcto, reply_mode correcto
- **Condition**: spam filtrado (va a OUTPUT), no-spam continua a RAG
- **RAG**: contexto recuperado (o vacio sin error)
- **Respondedor**: email generado con estructura plantilla, max 200 palabras
- **Gmail**: email enviado (verificar en bandeja de salida)

**Paso 4: Iterar si hay fallos**

Si un nodo falla:
1. Identificar la causa en el output/error del node_state
2. 90% de los casos: ajustar instructions del nodo en la BD directamente:
   ```sql
   UPDATE canvases SET flow_data = json_set(flow_data, '$.nodes[X].data.instructions', 'nuevas instrucciones') WHERE id = '{canvas_id}'
   ```
3. Re-ejecutar el piloto
4. Repetir hasta que los 3 emails completen el pipeline
5. Ultimo recurso: cambiar modelo del nodo y documentar que el problema era el modelo

**Paso 5: Registrar output final**

Guardar en una variable los outputs reales de cada nodo para usar en Task 3.
