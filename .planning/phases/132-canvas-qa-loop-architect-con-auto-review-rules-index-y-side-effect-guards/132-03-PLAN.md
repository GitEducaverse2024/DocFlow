<objective>
Cerrar el loop de calidad con guards automáticos + auto-reparación runtime:

1. Añadir `isSideEffectNode` + `insertSideEffectGuards` a canvas-flow-designer.ts como
   post-procesador del flow_data. Cada nodo destructivo recibe un condition guard + un
   agent reporter justo antes. Llamado desde `finalizeDesign` después de validateFlowData.

2. Crear canvas-auto-repair.ts con `attemptNodeRepair` que lee canvas_runs, llama al LLM
   con AGENT_AUTOFIX_PROMPT (ya existe por Plan 02), aplica el fix a flow_data si es
   posible, resetea node_states para re-ejecución, y trackea intentos en
   canvas_runs.metadata.repair_attempts para evitar loops infinitos.

3. Registrar `_internal_attempt_node_repair` como tool interno de catbot-tools.ts con
   gating name-based (prefijo `_internal_` filtrado en getToolsForLLM). El tool es
   autocontenido: recibe canvas_id + failed_node_id + guard_report como args, sin
   depender del contexto de ejecución.

4. Si el auto-repair falla por segunda vez: logKnowledgeGap + createNotification
   propagando channel_ref desde intent_jobs.

Purpose: Transformar fallos silenciosos en auto-reparación visible. Caso Holded Q1: si el
Maquetador emite html_body vacío, el guard antes del Gmail send detecta el fallo, CatBot
ajusta las instructions del Maquetador, re-ejecuta, y el email llega con contenido real.
Si tras el fix el guard vuelve a fallar, el usuario recibe un mensaje claro por Telegram.
