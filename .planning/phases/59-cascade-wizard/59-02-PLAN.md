**A. Create `app/src/components/tasks/pipeline-section.tsx`**

"use client" component that replaces the current step 3 content. Props:

```typescript
interface PipelineSectionProps {
  steps: PipelineStep[];
  setSteps: (steps: PipelineStep[]) => void;
  agents: Agent[];
  skills: Skill[];
  connectors: ConnectorInfo[];
  onFetchConnectors: (agentId: string) => void;

  onNavigateToCanvasNew: (stepIndex: number) => void;  // For WIZD-05
}
```

Implementation:
1. **Add step dropdown**: Replace the current 3-button row with a single "+" button that opens a dropdown (DropdownMenu or a simple popover). Options:
   - Agente (Bot icon) -- creates agent step (same as current)
   - Canvas (LayoutGrid icon from lucide) -- opens CanvasPickerSheet
   - Checkpoint (Shield icon) -- creates checkpoint step (same as current)
   - Sintesis/Merge (GitMerge icon) -- creates merge step (same as current)
   - Fork (GitFork icon from lucide) -- creates fork step group

2. **Canvas flow**:
   - When "Canvas" is selected, open CanvasPickerSheet
   - `onSelect`: create a canvas PipelineStep with `type: 'canvas'`, `canvas_id: selected.id`, `name: selected.name`, and store canvas metadata in a local Map for display
   - `onCreateNew`: save entire wizard state to localStorage key `wizard_draft_{Date.now()}`, then call `onNavigateToCanvasNew(currentStepIndex)` which navigates to `/canvas/new?from_task=draft&step_index={n}`
   - On page mount in the parent wizard, check for `from_task` search param. If present and localStorage has wizard draft, restore state and show a toast "Estado del wizard restaurado"

3. **Fork flow**:
   - When "Fork" is selected, generate a `fork_group` UUID
   - Create 3 PipelineStep entries: one with `type: 'fork'`, and one with `type: 'join'`, and initialize 2 branches (each as an array of steps stored in a local `forkBranches` Map keyed by fork_group)
   - Render ForkStepConfig between the fork and join steps in the pipeline list
   - When adding steps within a fork branch, those steps get the same `fork_group` and appropriate `branch_index`

4. **Existing step types**: Reuse the existing SortableStepCard component for agent/checkpoint/merge steps. Keep @dnd-kit drag-and-drop for reordering (but only for non-fork steps; fork groups are fixed position).

5. **Step rendering order**: Render steps linearly, but when encountering a `type: 'fork'` step, render the ForkStepConfig block (which includes all branch steps and the join) and skip ahead past the join.

6. **Summary for section collapse**: "{N} pasos en el pipeline" or "Pipeline vacio"
