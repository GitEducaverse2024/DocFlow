**Per-file specifics:**

1. `SE01-guard-before-emit.md` — title "SE01 — Condition guard antes de emit". Body: "Antes de cada `send`/`write`/`upload`/`create` el canvas inserta automáticamente un nodo guard de tipo condition que valida el contrato de entrada del emisor. Scope: emitter. **Por qué:** sin guard un connector puede emitir con contrato corrupto (campo vacío → email sin destinatario, Holded create sin funnelId). **Cómo aplicar:** Pipeline Architect añade guard con `required_fields: [...]` derivado del contrato. Si guard.false → ver SE03. **Relacionado:** SE02, SE03, R22 (RefCodes para lookup tolerante)."

2. `SE02-guard-validates-contract.md` — title "SE02 — Guard valida contrato completo". Body: "El guard valida que el contrato de entrada tiene TODOS los campos requeridos y ninguno está vacío/null. **Por qué:** validación parcial deja pasar edge cases (string vacío, array vacío) que rompen el emitter. **Cómo aplicar:** guard declara lista explícita de `required_fields`; check por field usa `isNonEmpty(v) = v != null && v !== '' && (Array.isArray(v) ? v.length > 0 : true)`. **Relacionado:** SE01, SE03, R10 (preservar campos entrantes)."

3. `SE03-guard-false-auto-repair.md` — title "SE03 — Guard.false → auto-repair 1 vez". Body: "Si guard.false, el canvas ruta a un agent reportador que pide a CatBot auto-reparar el input via `canvas_repair_input` una sola vez; si la segunda pasada sigue en guard.false, llama `log_knowledge_gap` y finaliza en error controlado. **Por qué:** infinite loops de auto-repair son peores que un fallo limpio + gap registrado. **Cómo aplicar:** nodo reportador con flag `max_retries: 1`; si retry.count > 1 → `{accion_final: 'no_action', reason: 'auto_repair_exhausted'}`. **Relacionado:** SE01, SE02, R24 (no-destructive fallback)."

4. `DA01-no-arrays-to-toolcalling.md` — title "DA01 — No arrays >1 item a tool-calling". Body: "No pases arrays con N>1 items directamente a nodos con tool-calling interno (CatPaw con tools, CatBrain+tool). Usa ITERATOR/Dispatcher. **Por qué:** el LLM del nodo tool-calling rara vez invoca N tool-calls correctos cuando el input es un array; success rate <40% en arrays >3 items. **Cómo aplicar:** si contrato del nodo recibe array → precedor DEBE ser ITERATOR con `items_from: input.array_field`. **Relacionado:** R02, R14."

5. `DA02-no-unused-connectors.md` — title "DA02 — No enlazar connectors/skills sin usar". Body: "No vincules a un nodo CatPaw connectors o skills que su system_prompt + instructions no usan. **Por qué:** cada tool extra añade ~150-300 tokens al context y reduce la probabilidad de que invoque el tool correcto. **Cómo aplicar:** revisar que TODOS los tools linkeados aparecen por nombre en el prompt del CatPaw o en las instructions del nodo. Si no: desvincula. **Relacionado:** R08, R15."

6. `DA03-no-llm-urls.md` — title "DA03 — No generes URLs con LLM". Body: "No uses un LLM para generar URLs (tracking, redirect, deep-link). Usa campos del output del tool emisor. **Por qué:** los LLMs confunden dominios, parámetros, encoding. Real bugs: URLs a `docatflow.com` en vez de `docflow.com`; query params perdidos. **Cómo aplicar:** el emitter devuelve su URL/ID; downstream nodes lo consumen por field-name. Si necesitás URL derivado, generalo con código (template string). **Relacionado:** R20, R21."

7. `DA04-no-implicit-dependencies.md` — title "DA04 — No dependas de datos fuera del input explícito". Body: "El nodo solo puede usar datos que llegan en su predecessorOutput. No asumas que el canvas preservó una variable del START o de un nodo lejano. **Por qué:** aunque R10 preserva campos, el CatBot puede perder campos no-canónicos. **Cómo aplicar:** cada nodo DEBE declarar explícitamente su contrato input y output. Si necesita un field del START, el pipeline lo propaga campo-a-campo. **Relacionado:** R10, R12, R13."
