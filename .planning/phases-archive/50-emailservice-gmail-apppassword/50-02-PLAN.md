**1. Extend POST /api/connectors for gmail** (EMAIL-09):
In `app/src/app/api/connectors/route.ts`:
- Import `encrypt` from `@/lib/crypto`
- After validation, if `type === 'gmail'`:
  - Extract gmail-specific fields from body: `user, account_type, auth_mode, from_name, app_password, client_id, client_secret, refresh_token`
  - Validate: `user` is required, `account_type` must be 'personal' or 'workspace', `auth_mode` defaults to 'app_password'
  - Build config object:
    - `user`, `account_type`, `auth_mode`, `from_name` (if provided), `client_id` (if provided) — stored in clear
    - `app_password_encrypted: encrypt(app_password.replace(/\s/g, ''))` if provided
    - `client_secret_encrypted: encrypt(client_secret)` if provided
    - `refresh_token_encrypted: encrypt(refresh_token)` if provided
  - Override the config in the INSERT with this encrypted config
  - Set `gmail_subtype` column: `'gmail_personal'` or `'gmail_workspace'` based on account_type
  - Set emoji to `'📧'` by default for gmail
- The INSERT statement needs to include `gmail_subtype` column

**2. Extend PATCH /api/connectors/[id] for gmail** (EMAIL-10):
In `app/src/app/api/connectors/[id]/route.ts`:
- Import `encrypt, isEncrypted` from `@/lib/crypto`
- In the PATCH handler, after loading the existing connector:
  - If connector type is 'gmail' and `body.config` is provided:
    - Parse existing config from DB
    - For each sensitive field (`app_password_encrypted`, `client_secret_encrypted`, `refresh_token_encrypted`):
      - If new value provided and NOT already encrypted (i.e., user sent plaintext): `encrypt(newValue)`
      - If new value provided and already encrypted: keep as-is (no double encryption)
      - If not provided in update: keep existing value from DB
    - Non-sensitive fields (user, account_type, auth_mode, from_name, client_id): update directly
    - Update `gmail_subtype` if `account_type` changed
  - Serialize merged config to JSON before saving

**3. Extend GET /api/connectors and GET /api/connectors/[id] for masking** (EMAIL-09):

  - Keep clear fields: `user`, `account_type`, `auth_mode`, `from_name`, `client_id`
  - Re-serialize config before returning
- Create a helper function `maskGmailConfig(connector)` to DRY this logic — can be in the route file itself or in a shared util
