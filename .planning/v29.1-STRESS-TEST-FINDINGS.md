# v29.1 Milestone — Stress Test Findings (Post-Phase-155)

## Context

Phase 155 closed with `status: passed` (12/12 must-haves). After closure, manual stress-testing was performed against CatBot via the chat API to validate the milestone claim: **"the KB is the single canonical source of DocFlow documentation, consumed by CatBot exclusively through search_kb + get_kb_entry + list_* tools"**.

Eight scenarios were tested across all six tracked entities (catpaws, skills, connectors, catbrains, email-templates, canvases), with full create → edit → delete cycles where tools existed. Evidence captured in chat tool-call logs + DB/KB state snapshots.

## Result

The milestone claim is **partially false**. Canvas operations completely bypass the KB. CRUD tool coverage is asymmetric (only `email_template` is complete). KB has systemic drift with 10 orphan files across 4 entities.

## Gap Inventory (8 gaps quantified)

### G1 — Canvas is invisible to KB (HIGH, blocking)

**Severity:** HIGH — directly contradicts milestone claim.

**Evidence:**
- `/api/canvas/route.ts` (POST): 0 `syncResource` calls
- `/api/canvas/[id]/route.ts` (PATCH, DELETE): 0 `syncResource` calls
- `catbot-sudo-tools.ts` `delete_catflow`: hard `DELETE FROM canvases`, no `markDeprecated`, no KB sync
- Test: created canvas via `canvas_create` tool → DB row appeared → **no** `.md` file created
- State: DB has 1 canvas, KB has 3 active `.md` files; all 3 are orphans (no DB match)

**Scope:**
- `knowledge-sync.ts` already supports `canvas` entity type (line 39, 82, 101, 110 — plumbing ready)
- Missing: hook calls in 3 API route handlers + `delete_catflow` sudo tool
- Estimated effort: 60 min

**Acceptance:**
- POST /api/canvas → `.md` file created in `resources/canvases/`
- PATCH /api/canvas/[id] → version bumped, `change_log` appended
- DELETE /api/canvas/[id] → `status: deprecated`, file preserved
- `delete_catflow` sudo → same soft-delete behavior as DELETE
- Creating canvas via CatBot's `canvas_create` tool triggers the above (tool calls the API)

### G2 — `create_catbrain` silently drops knowledge entries (HIGH, UX)

**Severity:** HIGH — user intent silently lost.

**Evidence:**
- Tool schema only accepts `{ name, purpose }` (catbot-tools.ts line ~300)
- Test: user prompted "create catbrain with 3 knowledge entries: [...]" → CatBot called `create_catbrain` with only `name + purpose`, ignored the 3 entries without acknowledgement
- Resulting catbrain has 0 entries in `catbrain_connectors` and 0 in any entry table

**Scope:**
- Research needed: catbrain data model (likely qdrant vector collection, not SQL)
- Add `entries` parameter to tool OR follow-up tool `add_catbrain_entry`
- Estimated effort: 2h (research + implement)

**Acceptance:** When user gives entries in prompt, they get stored OR CatBot explicitly acknowledges "I can't add entries, please use UI".

### G3a — `delete_cat_paw` tool missing (MEDIUM, symmetry)

**Severity:** MEDIUM — asymmetric with `create_cat_paw` / `update_cat_paw`.

**Evidence:** `grep delete_cat_paw` → only appears in `/api/cat-paws/[id]` route. No tool definition in catbot-tools.ts TOOLS array, no case in executeTool switch.

**Scope:** Add tool definition + case that calls DELETE /api/cat-paws/[id] (endpoint already has soft-delete via syncResource).

**Estimated effort:** 20 min.

**Acceptance:** CatBot can delete a CatPaw via chat (with confirmation prompt); KB file marked deprecated via existing API hook.

### G3b — Broader CRUD tool gaps (MEDIUM, product decision)

**Severity:** MEDIUM — UX incomplete but UI covers the gap.

**Missing tools:**
- `create_skill`, `update_skill`, `delete_skill` (skills have NO CatBot write tools at all)
- `update_connector`, `delete_connector`
- `update_catbrain`, `delete_catbrain`
- `canvas_update` (rename/mode change — node-level tools exist but canvas-level metadata cannot be edited)
- `update_task`, `delete_task`
- `delete_intent`

**Scope:** ~8-10 new tools. Requires product decision on scope.

**Estimated effort:** 3h.

**Acceptance:** Every entity has full CRUD symmetry via CatBot chat, OR explicit rationale for asymmetry documented.

**Recommendation:** Defer to Phase 156 (v29.2) pending product scope decision.

### G4 — Link tools don't re-sync parent CatPaw KB (MEDIUM)

**Severity:** MEDIUM — affects discoverability.

**Evidence:**
- `link_connector_to_catpaw` (catbot-tools.ts ~2122): inserts into `cat_paw_connectors`, no `syncResource('catpaw', 'update', ...)` call
- `link_skill_to_catpaw` (~2148): inserts into `cat_paw_skills`, no `syncResource` call
- Test: created CatPaw, linked Holded connector, linked Holded ERP skill → CatPaw `.md` file has no section listing linked connectors/skills
- Consequence: `search_kb({search: "holded connector"})` does NOT find the CatPaw via KB; requires DB tool `get_cat_paw` to see relations

**Scope:** Add `syncResource('catpaw', 'update', paw_row, hookCtx(...))` at end of both link tool cases. Also add `## Conectores vinculados` and `## Skills vinculadas` sections to the catpaw KB template in knowledge-sync.ts.

**Estimated effort:** 30-45 min.

**Acceptance:** After linking, CatPaw KB file shows linked connectors + skills; version bumped.

### G5 — KB↔DB drift: 10 orphan files (HIGH, hygiene)

**Severity:** HIGH — contradicts "KB as single canonical source" claim.

**Evidence (post-stress-test snapshot):**

| Entity | DB rows | KB active | Orphans |
|--------|---------|-----------|---------|
| catpaws | 38 | 44 | **6** |
| skills | 43 | 44 | **1** |
| email-templates | 16 | 17 | **1** |
| canvases | 1 | 3 | **2** |
| connectors | 12 | 12 | 0 ✓ |
| catbrains | 3 | 3 | 0 ✓ |

**Scope:** Run `scripts/kb-sync.cjs --audit-stale` to identify, then `--archive --confirm` to move orphans to `.docflow-legacy/orphans/`. Alternative: `--full-rebuild --source db` to regenerate all from DB ground truth.

**Estimated effort:** 15 min (ops).

**Acceptance:** Active-count per entity matches DB row count exactly.

### G6 — Poor auto-generated summary field (LOW, quality)

**Severity:** LOW — cosmetic but affects `search_kb` ranking.

**Evidence:** Auto-generated summaries for connector + catbrain literally say `"connector <uuid>"` and `"catbrain <uuid>"`. Should synthesize from DB `description` or `purpose` field.

**Scope:** `knowledge-sync.ts` summary synthesis — per entity, prefer `description → purpose → fallback to "<entity> <short_id>"`.

**Estimated effort:** 30 min.

### G7 — Inconsistent `updated_by` on delete (LOW, correctness)

**Severity:** LOW — breaks audit trail consistency.

**Evidence:**
- `DELETE /api/cat-paws/[id]`: `updated_by: api:cat-paws.DELETE` ✓ (correct)
- `delete_email_template` (via catbot-tools.ts): `updated_by: web:default` ✗ (wrong actor)

**Scope:** Trace `hookCtx` propagation in email-templates delete path. Apply same pattern as cat-paws.

**Estimated effort:** 10 min.

### G8 — Duplicate files with `-2` suffix (LOW, investigation)

**Severity:** LOW — unclear if bug or intentional versioning.

**Evidence:** Files like `155c955e-pro-educaverse.md` + `155c955e-pro-educaverse-2.md` (same prefix, `-2` suffix). Pattern repeats for `9f97f705-pro-revi`, `bc03e496-pro-k12`, `d7cc4227-pro-simulator`.

**Scope:** Investigate slug-collision handling in `knowledge-sync.ts`. Determine if `-2` is intentional version branching or a bug where slug generator retries on filesystem collision.

**Estimated effort:** 1h (investigation + fix if bug).
