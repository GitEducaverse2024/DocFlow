**Fecha creación:** 2026-04-20

## PROPÓSITO

Entregar el Knowledge Base (`.docflow-kb/`) como infraestructura de conocimiento **viva y consumida**, no solo como esqueleto populado. El milestone cierra cuando:

1. El KB contiene **todo** el conocimiento estructurado del proyecto (no solo los 66 recursos DB poblados en Phase 150).
2. **CatBot consume** el KB en cada sesión (prompt-assembler lee `_header.md`, tools `get_kb_entry`/`search_kb` operativas).
3. **Creation tools** (`create_cat_paw`, etc.) actualizan el KB automáticamente — sin paso manual.
4. **Dashboard `/knowledge`** permite explorar el KB desde el browser.
5. **Los dos knowledge layers legacy** (`.planning/knowledge/*.md`, `app/data/knowledge/*.json`) están eliminados — el KB es la única fuente canónica.

Fuente arquitectónica: `.planning/ANALYSIS-knowledge-base-architecture.md` §7 Fases 3-7.

## RIESGOS CONOCIDOS

1. **CatBot sin tools KB lo hace ciego** — hasta que Phase 152 termine, el KB poblado (66 archivos) es infraestructura sin consumidor. Cualquier fase que dependa del KB como oráculo de CatBot debe esperar a 152.

2. **Creation tools actuales duplican estado** — hasta que Phase 153 termine, crear un CatPaw genera row DB pero el KB queda stale. `kb-sync.cjs --full-rebuild --source db` compensa manualmente.

3. **Dos fuentes de verdad temporalmente** — entre ahora y Phase 155, `.planning/knowledge/*` y `.docflow-kb/` coexisten. Code que lea del layer antiguo sigue funcionando pero no aprovecha el KB.

4. **Test debt heredado** — 18 tests rojos (pre-existentes) en `knowledge-tree.test.ts`, `knowledge-tools-sync.test.ts`, `task-scheduler`, `alias-routing`, `catbot-holded-tools`. Documentados en deferred-items.md de Phases 150/151. Phase 155 los debe cerrar (pasar o borrar).
